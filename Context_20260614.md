## 3. La Bible Métier — Règles de Calcul

### 3.1 Structure d'une Garde Partagée
- **1 nounou** pour **2 familles** (A et B)
- **2 à 3 enfants** au total, max 2 par famille, min 1 par famille
- Les deux familles sont co-employeurs : chacune paie une part du salaire net + ses charges

### 3.2 Calcul du Salaire Net Mensuel
```
hNorm  = min(hHebdo, 40)
hSup25 = min(max(0, hHebdo - 40), 8)   → +25%  (40h à 48h/sem)
hSup50 = max(0, hHebdo - 48)            → +50%  (>48h/sem)

salNetMens = (hNorm × 52/12 × taux
            + hSup25 × 52/12 × taux × 1.25
            + hSup50 × 52/12 × taux × 1.50) × ratioPresence
```
**ratioPresence** = (joursOuvrables - joursMaladie) / joursOuvrables

### 3.3 Répartition entre Familles
- `repartitionA` (0 → 1) : part du salaire net payée par Famille A
- `repartitionB` = `1 - repartitionA`
- **Calcul mode B** (`calcBModeRepartition`) : ratio proportionnel aux heures de présence de chaque enfant

### 3.4 Indemnités (indépendantes du ratio salarial)
- `repartitionIndemA` (défaut 0.5) : part des indemnités pour Famille A, **indépendante** de `repartitionA`
- S'applique à : **Navigo + Frais km + Frais d'entretien**
- Frais d'entretien = `joursEntretienBase × tauxPresenceJour × indemEntretien × indemRatio`
- ⚠️ PAS de multiplication par `qp` (ratio salarial) — bug corrigé mai 2026

### 3.5 Reste à Charge (RAC) — Mode Magique vs Expert

#### Mode Magique (défaut)
Le système calcule automatiquement le ratio optimal pour équilibrer le RAC entre familles.

**CMG CAF 2025 (barème Pajemploi)**
| Revenus fiscaux | 1 enfant | 2 enfants |
|----------------|----------|-----------|
| ≤ 31 000 € | 1 075 €/mois | 1 612 €/mois |
| ≤ 57 000 € | 755 €/mois | 1 132 €/mois |
| > 57 000 € | 570 €/mois | 855 €/mois |

**Crédit d'impôt (CI) — plafond annuel**
- 1 enfant : 6 750 €/an → 562,50 €/mois
- 2+ enfants : 7 500 €/an → 625,00 €/mois
- Taux : 50% des dépenses nettes d'emploi

**Algorithme `calcEquitableRatioIteratif`** : balayage de 1% à 99% (990 itérations) pour trouver le `repartitionA` qui minimise l'écart RAC entre A et B.

#### Mode Expert (déverrouillé manuellement)
L'utilisateur saisit manuellement les 5 aides par famille :
`cmgCotisations`, `cmgRemuneration`, `abattementCharges`, `aideVille`, `creditImpot`

Pré-rempli depuis les estimations du Mode Magique. Reset disponible.

### 3.6 Congés & Absences
- `congesJson` sur le modèle Garde : config des congés payés
- `evenementsJson` sur Mois : liste des événements (CP, maladie nounou)
- Les congés payés ne réduisent PAS le salaire mensuel (provision mensuelle)
- La maladie de la nounou réduit le `ratioPresence`

## 4. Principes UX/UI

### 4.1 Progressive Disclosure (Divulgation Progressive)
**Principe cardinal** : ne montrer la complexité que quand l'utilisateur la demande.

- Étape 1 (acteurs) : uniquement les compteurs d'enfants — pas de noms
- Mode RAC : "Mode Magique" par défaut, bouton "⚙️ Ajuster manuellement" pour Mode Expert
- Settings : onglets séparés par domaine (acteurs, planning, paie, **personnaliser**)

### 4.2 Smart Defaults
Tout champ a une valeur par défaut sensée — l'utilisateur ne DOIT rien saisir pour avancer :
- Prénoms enfants : **Simone, Giselle, Suzanne** (charmants, mémorables, modifiables)
- Taux horaire : 11 €/h net
- Navigo : 90,80 €/mois
- Frais entretien : 6 €/jour
- Répartition indemnités : 50/50
- Répartition RAC : calculée automatiquement (Mode Magique)

### 4.3 Inputs Numériques — Pattern "Raw State"
Pour éviter les inputs incontrôlables (bug où taper "66" était impossible) :
```tsx
const [raw, setRaw]         = useState(() => String(Math.round(value * 100)));
const [focused, setFocused] = useState(false);
// Sync depuis parent seulement si pas focused
useEffect(() => {
  if (!focused) setRaw(String(Math.round(value * 100)));
}, [value, focused]);
```

### 4.5 Grammaire Couleur — Vue Mensuelle (juin 2026)


**Priorité de détection** (ordre décroissant) :
1. Jour hors du mois courant → `off`
2. Jour férié français → `holiday`
3. Événement `conge_paye` → `cp`
4. Événement `maladie_nounou` → `sick`
5. Sinon → `worked`

### Hook pre-commit ESLint (automatique)
Configuré dans `.claude/settings.json` (gitignored). Avant chaque `git commit`, Claude Code exécute automatiquement `npx next lint`. Si ESLint échoue, le commit est bloqué.

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "cmd=$(jq -r '.tool_input.command'); if echo \"$cmd\" | grep -qE '^git commit'; then cd /Users/arthurboueilh/Documents/GitHub/nannypay-front && npx next lint || exit 1; fi",
        "timeout": 90
      }]
    }]
  }
}
```
Dépendance : `jq` installé via `brew install jq`.

## 7. Points de Vigilance Récurrents

| Problème | Solution |
|----------|----------|
| Clerk v7 vs v5 | Toujours `@clerk/nextjs@^5.7.6` avec Next 14 |
| `NEXT_PUBLIC_*` | Inlinée au BUILD — doit être dans Vercel AVANT le build |
| Input numérique | Pattern `raw` + `focused` (pas de controlled input direct) |
| `repartitionIndemA` ≠ `repartitionA` | Indemnités et salaire se répartissent indépendamment |
| Prisma types | Après migration schema → `npx prisma generate` localement |
| SSG + Clerk | `ClerkProvider` gracieux si clé absente (guard `if (!PUBLISHABLE_KEY)`) |
| `tsc --noEmit` en local | Échoue car `@clerk/nextjs`, `svix` non installés en dev — NE PAS utiliser comme gate CI |
| ESLint `no-unused-vars` | Quand on supprime du JSX, nettoyer **en même temps** les imports, états, fonctions et props qui ne servent plus — sinon Vercel build échoue |
| `next lint` vs `tsc` | Vercel bloque sur ESLint (via `next build`), pas sur les erreurs TypeScript pures — vérifier avec `npx next lint` avant de committer |
